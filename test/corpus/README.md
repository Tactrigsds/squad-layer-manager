# Squad server corpus

Ground-truth samples of squad server output, used to keep the squad server emulator honest.

All committed corpus files have IPv4 addresses rewritten to the RFC 5737 documentation ranges
and RCON credentials redacted. The capture/extract scripts apply this at write time
(`src/scripts/anonymize-ips.ts`); if you copy a raw log window in manually (like
`ttscrim2-window-*.log`), run the anonymizer CLI over it before committing.
The emulator's rendered log lines and RCON responses should be validated against these samples
(and against the app's own matchers) so the two can't drift apart silently.

## logs/

`log-corpus.json` is generated by `src/scripts/extract-log-corpus.ts` from real `SquadGame.log`
files (not committed; kept in `./logs` of the main checkout). Contents:

- `eventCounts`: how often each `SM.LogEvents` matcher fired across the scanned files.
- `eventSamples`: raw log entries (including continuation lines) per matched event type.
- `chainExamples`: complete chainID groupings for the chain events (PLAYER_CONNECTED_CHAIN,
  ROUND_ENDED_CHAIN, PLAYER_KICKED_CHAIN), including the unmatched noise lines that share the
  chainID. This is what a faithful emulator should reproduce, not just the matched lines.
- `unmatchedFamilies`: frequency table of unmatched line families with samples, i.e. the
  background noise a real server emits. Families occurring fewer than 20 times are folded into
  an aggregate count.
- `preambles`: the headerless startup lines at the top of each log file (the receiver/parser
  must skip these, so the emulator should emit them).
- `parseErrors`: entries where a matcher regex hit but `onMatch`/schema validation failed.

Source files scanned (2026-07-12): `SquadGame-backup-2026.07.03-06.55.05.log`,
`SquadGame-backup-2026.07.06-09.40.44.log`, `SquadGame.log`, `tiger.log`, `onlytiger.log`.

### Known gaps / findings

- `LAYER_CHANGED` matched 0 times across ~160MB of logs. Its regex may be stale against current
  Squad versions; verify against a live server before implementing it in the emulator.
- RCON chat-stream events (CHAT_MESSAGE, PLAYER_WARNED, POSSESSED_ADMIN_CAMERA, SQUAD_CREATED,
  PLAYER_BANNED, SQUAD_RENAMED) arrive over RCON, not the log file, so they are not in
  `log-corpus.json`; they are captured live (see below).
- The server closes older RCON sessions when a client attempts multiple logins from the same IP
  (`LogRCONServer: Warning: ...: Client is attempting multiple logins from <ip>, stopping older
  sessions`). The emulator should reproduce this.

## rcon/

`capture-*.json` files are produced by `src/scripts/capture-rcon-corpus.ts` run against a live
server (dev/test server over tailscale). They record every raw packet in both directions (hex +
decoded fields), per-command packet groupings, assembled response bodies, and passively received
chat-stream (type 1) packets. This is the framing-level ground truth for the emulator's RCON
frontend: exact packet ids, empty-terminator behavior, multi-packet responses (`ListLayers`,
`ShowServerInfo`), and error shapes for bogus commands.

Usage:

```sh
set -a; source .env.rcon; set +a
pnpm run script src/scripts/capture-rcon-corpus.ts [--mutate] [--listen-secs N]
```

`--mutate` also runs harmless mutating commands (broadcast, warn, set next layer variants).
`--listen-secs N` keeps the connection open to passively capture chat packets, e.g. while a
player performs the in-game checklist below.

### Findings from the 2026-07-12 live capture (tt-scrim-2)

Framing (see the capture JSONs for raw bytes):

- Response to command id N: data packets with id N, then two empty responses with the probe's
  id N+2, then a raw 7-byte sequence `00 01 00 00 00 00 00`. core-rcon decodes that 7-byte
  sequence into a pseudo-packet with body `\x01`, which is what actually triggers response
  assembly (`response${id - 2}`). The emulator must reproduce this sequence exactly.
- Unknown commands return no data packets at all (just the terminator sequence), not an error.
- `AdminSetNextLayerWorkflow <bogus>` is silent; `AdminSetNextLayer <bogus>` returns
  `ERROR: Unable to set next layer : layer Not Found : <name>`.
- Missing args produce `Missing argument 0: (NameOrEOSId). Use "ShowCommandInfo ..." ...`.
- `AdminBroadcast` confirms with `Message broadcasted <msg>`.
- `AdminWarn "0"` (numeric playerID) fails with `Could not find player 0` even while player 0 is
  connected; the command takes NameOrEOSId only. Note the app sends
  `AdminWarn "${normalizeToPlayerId(ids)}"`.
- Command error feedback (e.g. `Could not find player 0`) is also echoed as a type-1
  chat-stream packet, not just as the command response.
- `AdminSetNextLayer <layer>` without factions auto-picks factions (ShowNextMap then reports
  e.g. `factions USA GFI`).
- Usernames may carry a leading space (no clan tag): `Name:  grey275` in ListPlayers,
  `]  grey275 :` in chat packets.
- The server logs join requests including the server password in plaintext
  (`?Name=grey275?PASSWORD=...`) in SquadGame.log.

`ttscrim2-window-2026-07-12.log` in `logs/` is the SquadGame.log window correlated with
`rcon/capture-2026-07-12T20-41-03-630Z.json` (same server, same time window: mutate battery plus
a player performing the checklist below). Fetched via `src/scripts/fetch-log-tail.ts`.

From the follow-up capture (`capture-2026-07-12T20-58-19-152Z.json`, same window in
`ttscrim2-window-2026-07-12.log`):

- `AdminWarn` works by EOS id and by bare name (matched despite the name's leading space). The
  confirmation `Remote admin has warned player  grey275. Message was "..."` arrives as the
  command response AND as a type-1 chat packet AND as a
  `LogSquad: ADMIN COMMAND: ... from RCON` log line.
- `AdminRenameSquad <team> <squadIndex>` takes no name argument; it resets the squad name to the
  default (`"A SQUAD"` -> `"Squad 1"`). The rename line appears twice in the command response
  body, once on the chat stream, once in the log.
- `AdminChangeLayer <layer>` responds `Change layer to <layer>`; the actual map travel is async
  (ShowCurrentMap still reports the old layer immediately after). Log side:
  `ADMIN COMMAND: Change layer to <layer> from RCON` (LAYER_CHANGED matches fine; it had just
  never been used in the older logs), then SeamlessTravel/TransitionMap/Bringing World lines.

Still missing real samples (gaps to fill later): `CHAT_MESSAGE` on ChatAll,
`POSSESSED/UNPOSSESSED_ADMIN_CAMERA` (player lacked in-game camera perms), `PLAYER_BANNED`,
self-teamswap (produced no distinct log line in the window).

### In-game capture checklist (single player)

Run the capture with `--mutate --listen-secs 300` while the player:

1. sends a chat message in each channel: all, team, squad (needs a squad), admin
2. creates a squad (SQUAD_CREATED), renames it via admin chat or console, leaves it
3. enters and exits admin camera (POSSESSED/UNPOSSESSED_ADMIN_CAMERA)
4. changes team and back
5. (optional, reconnect needed) disconnects and reconnects for a fresh join chain

RCON-side during the same window (the script's `--mutate` battery, plus manually if needed):
warn the player, broadcast, set next layer, force team change, disband squad, remove from squad,
kick the player last.
